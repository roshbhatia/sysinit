## ADDED Requirements

### Requirement: Canonical recipe set covers the openspec workflow
`~/.config/goose/recipes/` MUST contain at minimum the following recipe files, generated by `goose.nix`: `openspec-propose.yaml`, `openspec-apply.yaml`, `openspec-archive.yaml`, `openspec-explore.yaml`. Each recipe wraps the same workflow described in the corresponding Claude Code skill at `~/.claude/skills/openspec-<verb>/SKILL.md`. The recipe's `prompt` MUST embed the workflow instructions verbatim from a single source-of-truth Nix string (no duplicated text).

#### Scenario: Recipes are present after activation
- **WHEN** `nh darwin switch .` completes
- **THEN** `goose recipe list` (or equivalent) reports the four named recipes

#### Scenario: Recipe content matches the skill
- **WHEN** `~/.config/goose/recipes/openspec-propose.yaml` and `~/.claude/skills/openspec-propose/SKILL.md` are both inspected
- **THEN** the recipe's `prompt` (or `instructions`) field contains the same workflow text as the skill's body (modulo YAML escaping)

### Requirement: Recipes are read-only at the workflow level
No goose recipe shipped by this change SHALL invoke a mutating shell command without an explicit user confirmation step in the recipe definition. The forbidden commands include `nh darwin switch`, `nh os switch`, `git push`, `git commit` (recipes MAY stage and propose messages), `openspec archive` (recipes MAY validate/status), and any `rm`/`mv` outside scratch directories.

#### Scenario: Recipe inspects state but does not mutate
- **WHEN** the `openspec-explore` recipe runs
- **THEN** it invokes `openspec list`, `openspec status`, and `openspec instructions` only â€” no mutating subcommands

#### Scenario: Recipe attempts to bypass the read-only rule
- **WHEN** a contributor adds `nh darwin switch .` to a recipe's command list
- **THEN** code review flags it (no automated lint enforces this; it is a discipline-level constraint documented in goose.nix comments)

### Requirement: Recipe shape follows Goose's documented YAML schema
Each `<name>.yaml` MUST conform to Goose's recipe schema: top-level `name`, `description`, `instructions` (or `prompt`), and optionally `parameters` and `mcp_servers`. The home-manager-generated YAML SHALL pass `goose recipe validate <path>` (or whatever the equivalent Goose CLI validator is).

#### Scenario: Generated YAML parses
- **WHEN** `python3 -c 'import yaml; yaml.safe_load(open("~/.config/goose/recipes/openspec-propose.yaml"))'` is run after activation
- **THEN** parsing succeeds and the resulting object has at minimum the keys `name`, `description`, and `instructions` (or `prompt`)

#### Scenario: Missing required field
- **WHEN** a recipe is generated without a top-level `name`
- **THEN** `goose recipe list` (or equivalent) reports the file as malformed or silently skips it, AND a build-time assertion in `goose.nix` fails citing the missing field
